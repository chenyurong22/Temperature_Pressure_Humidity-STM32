# STM32 IoT Telemetry Client

Firmware for the ST B-L475E-IOT01A board. The device connects to a 2.4 GHz Wi-Fi network, reads onboard environmental sensors, and periodically sends telemetry to an HTTP API behind Traefik.

The main use case is to keep a remote server informed that the home network is alive. Because every API call reaches the server through the Livebox public IP, the backend can infer and log the current public IP address without the STM32 needing to know it.

## What It Does

- Initializes the STM32 HAL and board peripherals generated by STM32CubeIDE.
- Connects the ISM43362 Wi-Fi module to a configured 2.4 GHz SSID.
- Reads onboard temperature, humidity, and pressure sensors.
- Applies a configurable temperature offset to compensate for board self-heating.
- Builds a structured JSON payload with telemetry, runtime counters, network metadata, and firmware information.
- Sends the payload with an HTTP `POST` request to the configured API endpoint.
- Logs boot, Wi-Fi, telemetry, and API status over UART.
- Runs an independent watchdog and self-healing recovery loop so the board can reboot if firmware or Wi-Fi operations get stuck.

## Hardware

Target board:

- ST B-L475E-IOT01A Discovery kit
- STM32L475 MCU
- ISM43362 Wi-Fi module
- HTS221 humidity and temperature sensor
- LPS22HB pressure sensor

This project currently includes BSP drivers for temperature, humidity, and pressure only. The board has additional sensors physically present, but their BSP drivers are not part of this workspace yet.

## Project Structure

Application code lives in `Core/Inc` and `Core/Src`:

- `app.c` / `app.h`: high-level application loop, scheduling, counters, and retry behavior.
- `app_config.h`: versioned application configuration.
- `app_secrets.h`: local Wi-Fi credentials, ignored by Git.
- `app_secrets.example.h`: safe template for local secrets.
- `telemetry.c` / `telemetry.h`: sensor initialization and averaged readings.
- `wifi_service.c` / `wifi_service.h`: small wrapper around ST's `wifi.h` API.
- `api_client.c` / `api_client.h`: JSON payload creation and HTTP request sending.
- `log.c` / `log.h`: UART logging helpers.
- `system_health.c` / `system_health.h`: reset reason capture, independent watchdog startup, watchdog refresh, and forced MCU reset.
- `main.c`: STM32CubeIDE-generated initialization plus calls to `App_Init()` and `App_Loop()`.

## Configuration

Versioned configuration is in `Core/Inc/app_config.h`.

Important values:

```c
#define APP_DEVICE_ID "stm32-room"
#define APP_FIRMWARE_VERSION "1.2.1"
#define APP_LOCATION "room"

#define APP_API_HOST "iot.mathislambert.fr"
#define APP_API_IP_ADDR {82U, 67U, 120U, 109U}
#define APP_API_PATH "/api/ingest?device=" APP_DEVICE_ID
#define APP_API_PORT 80U

#define APP_SEND_INTERVAL_MS 300000UL
#define APP_SEND_RETRY_INTERVAL_MS 60000UL
#define APP_WIFI_RECOVERY_FAILURE_THRESHOLD 3U
#define APP_SYSTEM_RESET_FAILURE_THRESHOLD 12U
#define APP_TEMPERATURE_OFFSET_C 6.0f
```

The firmware uses a static API IP address to avoid DNS issues observed with the Wi-Fi module. It still sends the HTTP `Host` header, so Traefik can route by hostname.

## Local Secrets

Wi-Fi credentials must be stored locally in:

```text
Core/Inc/app_secrets.h
```

That file is ignored by Git. Use `Core/Inc/app_secrets.example.h` as a template:

```c
#ifndef APP_SECRETS_H
#define APP_SECRETS_H

#define APP_WIFI_SSID "your-2.4ghz-ssid"
#define APP_WIFI_PASSWORD "your-wifi-password"

#endif
```

Do not commit `Core/Inc/app_secrets.h`.

## Build And Flash

Recommended workflow:

1. Open the project in STM32CubeIDE.
2. Create `Core/Inc/app_secrets.h` from `Core/Inc/app_secrets.example.h`.
3. Build the project.
4. Flash the board with Run or Debug.
5. Open a serial terminal on the ST-LINK COM port at `115200 8N1`.

The local command-line build used during development was:

```powershell
make -C Debug all
```

The exact `make` and `arm-none-eabi-gcc` paths may depend on the STM32CubeIDE installation.

## Self-Healing Behavior

The firmware is designed to keep running unattended:

- The independent watchdog (IWDG) starts early during boot and uses the STM32L4 LSI clock with the maximum practical timeout, about 32 seconds.
- The watchdog is refreshed only from the main application loop. If the firmware blocks inside Wi-Fi, SPI, HAL, a fault handler, or `Error_Handler()`, the board resets automatically.
- Consecutive failures are tracked and reported in the API payload.
- After `APP_WIFI_RECOVERY_FAILURE_THRESHOLD` consecutive failures, the firmware closes the socket, disconnects Wi-Fi, resets the Wi-Fi module, and reconnects cleanly.
- After `APP_SYSTEM_RESET_FAILURE_THRESHOLD` consecutive failures, the firmware requests a full MCU reset with `NVIC_SystemReset()`.
- On boot, reset flags are captured before being cleared, then the reset reason is sent in the next payload.

## Notes

- The board Wi-Fi module connects to 2.4 GHz networks. A 5 GHz-only SSID will not work.
- HTTPS/TLS on port `443` was not reliable with the current Wi-Fi wrapper and module firmware. The working setup uses HTTP on port `80` and relies on the backend network configuration for exposure and routing.
- Build outputs under `Debug/` and `Release/` are intentionally ignored.
